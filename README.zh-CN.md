# CPA Control Center

面向 CPA 管理的 Codex auth 号池桌面运维工具。

`CPA Control Center` 把现有 CPA 管理接口收拢成一个专门的桌面应用，适合不想继续依赖浏览器页面、`localhost` 或终端脚本来维护大规模 auth 池的场景。

你只需要填写 `Base URL` 和 `Management Token`，就可以在一个原生窗口里完成库存同步、扫描、维护、日志、历史和导出。

## 借鉴来源与适用后端

- 本项目明确借鉴了 [`fantasticjoe/cpa-warden`](https://github.com/fantasticjoe/cpa-warden) 的思路和工作流设计。
- 本桌面工具当前主要面向 [`router-for-me/CLIProxyAPI`](https://github.com/router-for-me/CLIProxyAPI) 作为 CPA 后端使用。

## 概览

- 基于 Wails、Go、Vue 3、TypeScript 的原生桌面应用
- 只需要 `Base URL` 和 `Management Token`
- 首次连接优先做 inventory 同步，适合大号池
- 账号列表和扫描详情都已支持后端分页
- 统一状态模型：`待探测`、`正常`、`401 失效`、`限额`、`可恢复`、`错误`
- 一键扫描、一键维护
- 实时任务日志与扫描历史
- 支持 `401 失效` 和 `限额` 结果导出为 CSV / JSON
- 内置中英文双语界面

## 适合谁用

如果你符合下面这些情况，这个项目就是为你准备的：

- 已经部署了 CPA，并开启了管理接口
- 需要维护一个以 Codex 为主的 auth 号池
- 希望用桌面应用，而不是再额外部署一个后台
- 希望把扫描、维护、日志、导出集中到一个工具里

如果你的需求是这些，它暂时不是重点方向：

- 在 GUI 里新建或导入 auth 文件
- 在桌面端内做 OAuth 登录流程
- 做一个覆盖 CPA 全部能力的通用管理后台

## 它解决什么问题

大号池通常不是“完全不可用”，而是逐渐进入混合状态：

- 有些号已经 `401 Invalid`
- 有些号触发了额度限制
- 有些号探测失败，但再次探测又能恢复
- 有些号之前被禁用过，现在已经可以恢复

这个应用的目标只有两个：

1. 让你快速看清当前号池健康度。
2. 让你基于最新扫描结果执行可重复、可控的维护动作。

## 当前工作流

当前版本的使用流程被刻意拆成了几个阶段：

1. 保存 CPA 连接设置。
2. 先同步 inventory 到本地。
3. 在未完成首次扫描之前，也能直接看到已跟踪账号。
4. 需要健康状态时再执行扫描。
5. 确认最新扫描结果后再执行维护。

这样做的原因很直接：大号池首次连接时，不再需要一上来就探测几千个账号，界面也不会因为“尚未扫描”而完全空白。

## 首次连接后的行为

现在点击 **测试并保存** 成功后，应用会先进行一次轻量 inventory 同步：

- 从 CPA 拉取 auth 文件清单
- 建立本地库存快照
- 把新同步到但尚未探测的记录标成 `待探测`
- 让仪表盘和账号列表在首次扫描前也可用

这意味着：

- 仪表盘可以立刻显示已跟踪数量
- 账号列表可以立刻查看
- 完整健康状态要等首次扫描完成后才建立

## 状态模型

应用统一使用一套紧凑状态模型：

- `待探测`
- `正常`
- `401 失效`
- `限额`
- `可恢复`
- `错误`

其中 `待探测` 很关键。它表示“本地已经同步到库存，但还没有完成探测”，避免把这类账号误判成异常或空数据。

## 核心能力

### 1. 连接 CPA

只需要填写：

- `Base URL`
- `Management Token`

连接测试本身不会触发扫描。

### 2. 同步 Inventory

inventory 同步会：

- 从 CPA 拉取 auth 元数据
- 建立本地跟踪池
- 为大号池启动做准备
- 避免首次使用时仪表盘完全空白

### 3. 扫描号池

点击 **立即扫描** 后，应用会：

1. 从 CPA 重新拉取 auth inventory
2. 按 `targetType` 和 `provider` 过滤
3. 并发探测匹配账号
4. 写入最新本地快照
5. 生成一条扫描历史记录

### 4. 执行维护

点击 **执行维护** 后，应用会先重新扫描，然后按设置执行：

- 删除 `401 失效`
- 对 `限额` 账号执行 `禁用` 或 `删除`
- 对 `可恢复` 账号重新启用

所有破坏性动作都需要先确认。

### 5. 查看历史与日志

应用会保留：

- 当前账号快照
- 最近扫描历史
- 分页扫描详情
- 实时任务日志和进度事件

如果在设置中开启 **详细日志**，任务日志里还会显示逐账号探测和逐账号维护信息。

### 6. 导出问题账号

你可以导出当前：

- `401 失效` 账号
- `限额` 账号

支持格式：

- JSON
- CSV

## 大号池说明

当前实现已经针对几千账号规模做过一轮重构：

- 首次连接优先同步 inventory，而不是立即全量探测
- 仪表盘不再把全量账号传到前端
- 账号列表改成后端分页
- 扫描详情改成后端分页
- 已同步但未探测的账号会稳定保持 `待探测`

这还不是最终的性能上限，但已经明显比早期“前端全量快照”模型稳得多。

## 使用到的 CPA 接口

当前应用只依赖少量管理接口：

- `GET /v0/management/auth-files`
- `POST /v0/management/api-call`
- `DELETE /v0/management/auth-files?name=...`
- `PATCH /v0/management/auth-files/status`

健康探测通过 CPA 转发请求：

- `https://chatgpt.com/backend-api/wham/usage`

## 默认行为

| 配置项 | 默认值 |
| --- | --- |
| 语言 | 归一化后的系统语言（`en-US` / `zh-CN`） |
| 目标类型 | `codex` |
| 探测并发 | `40` |
| 动作并发 | `20` |
| 超时 | `15s` |
| 重试次数 | `1` |
| 限额处理 | `disable` |
| 删除 401 | 开启 |
| 自动恢复可恢复账号 | 开启 |
| 详细日志 | 关闭 |

## 重试模型

当前重试分成两层：

- 请求层重试：处理外层请求失败和 CPA 的瞬时错误，如 `408`、`429`、`5xx`
- 探测层重试：处理可恢复的探测异常，例如临时上游异常、返回体不完整、可重试状态码

不会盲目重试的情况包括：

- `401 Invalid`
- `Quota Limited`
- 明确缺少账号元数据

## 本地数据存储

应用会把本地状态保存在系统用户配置目录下的：

`CPA Control Center/`

目录通常包含：

- `settings.json`
- `state.db`
- `app.log`
- `exports/`

当前实现会保留最新快照，以及最近 `30` 次扫描历史。

## 安全建议

- 这是运维工具，不是演示页面；维护动作可能删除或禁用账号。
- 执行维护前，最好先看一遍最新扫描结果。
- 如果你在验证新的 CPA 环境，建议先关闭 `delete401`。
- 详细日志很适合排障，但在大号池下会更嘈杂。

## 当前边界

当前项目聚焦在：

- 管理已有 CPA auth 文件
- Codex 号池健康探测与维护
- Windows 优先的桌面体验，同时补齐 macOS 构建链路

当前不包含：

- auth 导入向导
- 应用内登录 / OAuth 获取
- 多节点统一编排
- 超出本地快照和历史视图之外的高级分析能力

## FAQ

### 会打开浏览器吗？

不会。它是 Wails 桌面应用，会直接打开自己的原生窗口。

### 它是完整的 CPA 后台吗？

不是。它是一个聚焦 auth 号池健康和维护的桌面运维工具。

### 为什么还没扫描就能看到已跟踪账号？

因为现在会先做 inventory 同步。这样可以先看到已跟踪库存，而健康状态会在首次扫描完成后从 `待探测` 变成真实探测结果。

### “执行维护”会先扫描吗？

会。维护一定会先跑一次新的扫描，再应用维护规则。

### 扫描详情支持大结果集吗？

支持。扫描详情已经改成后端分页，不会一次性把所有数据都塞进抽屉。

### 有没有调试模式？

有，但默认隐藏。按 `Ctrl + Shift + D` 可以打开内部调试面板，用于排查启动和仪表盘问题。

### 支持 macOS 吗？

构建链路已经补齐。正式的 macOS 产物建议在 Mac 或 `macos-latest` GitHub Actions runner 上生成。
