# CPA Control Center

面向 CPA 管理的 Codex auth 号池桌面运维工具。

这个项目的出发点很直接：当 auth 池规模上来以后，只靠浏览器页签或手动访问 `localhost` 去做管理，体验很割裂，也很容易漏看异常。`CPA Control Center` 把现有 CPA 管理接口包装成一个专注的桌面应用，无需进行任何部署，让你只需要连上 CPA，就能完成扫描、查看异常、执行维护和导出结果。

## 借鉴来源与适用后端

- 本项目明确借鉴了 [`fantasticjoe/cpa-warden`](https://github.com/fantasticjoe/cpa-warden) 的思路与工作流设计。
- 本桌面工具面向 [`router-for-me/CLIProxyAPI`](https://github.com/router-for-me/CLIProxyAPI) 这一 CPA 后端仓库使用，应用当前调用的管理接口也以它为主要适配目标。

## 概览

- 基于 Wails、Go、Vue 3、TypeScript 的原生桌面应用
- 只需要填写 `Base URL` 和 `Management Token`
- 通过 CPA 管理接口扫描 Codex auth 池
- 自动归类为 `正常`、`401 失效`、`限额`、`可恢复`、`错误`
- 执行维护前有确认弹窗，避免误操作
- 支持扫描历史、分页详情、实时任务日志、CSV/JSON 导出
- 内置中英双语界面

## 适合谁用

如果你符合下面这些情况，这个项目就是为你准备的：

- 已经部署了 CPA，并开启了管理接口
- 需要维护一个以 Codex 为主的 auth 号池（后续可能会进行其他渠道的维护）
- 希望用桌面应用，不想进行任何额外的部署，需要点开即用
- 希望把扫描、维护、日志、导出放到同一个工具里

如果你的需求是下面这些，那它暂时不是重点方向：

- 在 GUI 里新建或导入 auth 文件
- 在桌面端内做 OAuth 登录流程
- 需要一个覆盖 CPA 全部能力的通用管理后台

## 它解决什么问题

大号池的问题通常不是“完全不可用”，而是会出现各种混合状态：

- 有些号已经 `401 Invalid`
- 有些号碰到了额度上限
- 有些号之前被禁用过，但现在已经可恢复
- 有些号探测失败只是临时网络或上游异常

这个应用的核心目标只有两个：

1. 让你快速看清当前号池健康度。
2. 让你基于最新扫描结果执行可重复、可控的维护动作。

## 它能做什么

### 1. 连接 CPA

你只需要填写：

- `Base URL`
- `Management Token`

应用支持先测试连接再保存，而且“测试连接”本身不会触发扫描。

### 2. 扫描号池

点击 **立即扫描** 后，应用会：

1. 从 CPA 拉取完整 auth 清单
2. 根据你设置的 `targetType` 和 `provider` 做过滤
3. 并发探测匹配账号
4. 写入本地当前快照
5. 记录一条历史扫描记录，方便之后查看

### 3. 统一状态判定

扫描结果会被标准化成一组实际运维里足够好用的状态：

- `正常`
- `401 失效`
- `限额`
- `可恢复`
- `错误`

Dashboard、账号列表、维护逻辑和导出能力都基于这套状态。

### 4. 执行维护

点击 **执行维护** 后，应用会先跑一遍新的扫描，然后按你的设置处理：

- 删除 `401 失效`
- 对限额账号执行 `禁用` 或 `删除`
- 对可恢复账号执行自动重新启用

所有破坏性动作都需要先确认。

### 5. 查看历史与日志

应用会保留：

- 当前账号快照
- 最近扫描历史
- 分页扫描详情
- 实时任务日志和进度事件

如果你在设置里开启了“详细日志”，就能看到逐账号探测和逐账号维护日志。

### 6. 导出问题集合

你可以把当前的：

- `401 失效` 账号
- `限额` 账号

导出为：

- JSON
- CSV

## 核心能力

- 桌面优先的运维工作流
- Dashboard 健康概览与最近扫描记录
- 实时账号表，支持搜索、筛选、分页
- 单账号探测、禁用/启用、删除
- 任务日志中实时显示扫描/维护进度
- 对可恢复的临时探测错误支持自动重试
- 本地持久化：`settings.json`、`state.db`、`app.log`
- 支持 Windows 构建，也已补齐 macOS 构建流水线
- GitHub Actions 自动构建，并支持 tag 自动发布 Release

## 实际工作流

如果你只想用最短路径理解它，流程其实很简单：

1. 保存 CPA 连接信息。
2. 点击 **立即扫描**。
3. 看 Dashboard 和最近扫描记录。
4. 如有需要，打开扫描详情检查具体账号。
5. 当最新扫描结果符合预期时，点击 **执行维护**。
6. 需要交给别的系统或人工处理时，导出 401 / 限额结果。

## 使用到的 CPA 接口

这个应用刻意保持收敛，只依赖少量 CPA 管理接口：

- `GET /v0/management/auth-files`
- `POST /v0/management/api-call`
- `DELETE /v0/management/auth-files?name=...`
- `PATCH /v0/management/auth-files/status`

其中号池健康探测通过 CPA 代理请求：

- `https://chatgpt.com/backend-api/wham/usage`

## 默认行为

应用默认配置如下：

| 配置项 | 默认值 |
| --- | --- |
| 语言 | 归一化后的系统语言（`en-US` / `zh-CN`） |
| 目标类型 | `codex` |
| 探测并发 | `40` |
| 动作并发 | `20` |
| 超时 | `15s` |
| 重试次数 | `1` |
| 限额处理 | `disable` |
| 删除 401 | 开启 |
| 自动恢复可恢复账号 | 开启 |
| 详细日志 | 关闭 |

## 重试机制

当前重试分成两层：

- 请求层重试：针对外层请求失败，以及 CPA 返回的瞬时错误（`408`、`429`、`5xx`）
- 探测层重试：针对可恢复的探测异常，例如上游临时异常、返回结构不完整等

不会盲目重试的情况包括：

- `401 Invalid`
- `Quota Limited`
- 明确缺失账号元数据

## 本地数据存储

应用会把本地状态放到系统用户配置目录下的：

`CPA Control Center/`

目录内通常包含：

- `settings.json`
- `state.db`
- `app.log`
- `exports/`

当前实现会保留最新快照，以及最近 `30` 次扫描历史。

## 安全建议

- 这是运维工具，不是演示页面；维护动作可能删除或禁用账号。
- 执行维护前，最好先看一遍最新扫描结果。
- 如果你在验证新的 CPA 环境，建议先关闭 `delete401`，只观察扫描结果。
- 大号池下开启详细日志会更利于排障，但日志量也会明显增加。

## FAQ

### 它会打开浏览器吗？

不会。它是 Wails 桌面应用，会直接打开自己的原生窗口。

### 它是不是一个完整的 CPA 全功能后台？

不是。它是一个聚焦 auth 池健康和维护的桌面运维工具。

### “执行维护”会先扫描吗？

会。执行维护一定从一次新的扫描开始，然后再套用你的维护规则。

### 扫描详情支持大结果集吗？

支持。扫描详情已经改成后端分页，不会一次性把所有记录都塞进前端抽屉。

### 支持 macOS 吗？

已经补齐构建链路。正式 mac 产物建议在 Mac 或 `macos-latest` GitHub Actions runner 上生成。
